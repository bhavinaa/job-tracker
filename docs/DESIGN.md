## Revision History

- **v0.1 (2026-04-23, morning):** Initial approved design — "pluggable pull + tracker." Problem framed as multi-tool fragmentation; v1 wedge was "pull from a few RSS/Greenhouse sources + manual URL paste + CRUD tracker with LLM fit-score."

Tracking is still important but secondary. v1 wedge shifts from "feed-centric aggregation" to "company-centric polling from a user-curated target list." See the Recommended Approach for the v0.2 design; the v0.1 premises and architecture remain broadly valid with the listed additions.
- **v0.3 (2026-04-23, afternoon, later):** Cold-outreach as a first-class v1 path. Many target companies do not post publicly, or don't post the roles the user wants. The user's real workflow includes proactive cold email to named contacts. v0.3 adds: (a) `ats: cold-outreach` company type in `target_companies.yml`, (b) a `recipients` table tracking named contacts per company, (c) a cold-email drafter (distinct prompt from follow-up drafter), (d) new state-machine states for cold outreach that converge back into the normal application flow after a positive reply. Recipient identification stays manual in v1 (user pastes name/role/email); no LinkedIn scraping. LLM-assisted recipient research is deferred to v1.x.

## Problem Statement (v0.2)

The user is actively searching for jobs and internships across multiple buckets that don't share a single aggregator:

- **Big-tech MNCs with offices in SG and/or NYC** — TikTok/ByteDance, Meta, Google, Apple, Amazon, Microsoft
- **Startups in NYC** — predominantly YC / Greenhouse / Lever backed
- **Startups in Singapore** — smaller market, different boards (NodeFlair, MyCareersFuture) + LinkedIn
- **Cross-domain** — SWE, data, ML, PM, design — user is not a single-role applicant

No existing job board covers this Venn diagram well. LinkedIn is the closest but floods the user with irrelevant postings and doesn't surface opportunities at companies they haven't thought to search for. The pain is **not** "I applied to 100 things and can't track them" — it is **"I don't know what to apply to, and every company's jobs live on a different page."**

The v0.2 tool's job: let the user curate a target list of ~20–50 companies across their buckets, fetch jobs from each company's ATS directly, LLM-rank them against the user's profile, and surface a daily/weekly "new & matched" digest. Tracking is still built in (the user said "I need both"), but it stops being the primary framing.

## Demand Evidence

- **Scratch-your-own-itch.** The user IS the user. They are currently running a live job search with the multi-tool stack described above. Their pain is real, daily, and verifiable by dogfooding.
- **No externally verified demand beyond self.** When pushed to name three friends with the same specific pain, the user honestly admitted: "I'm generalizing — I don't have one specific story." This is a strong honesty signal but a weak market signal. Calibrated: this is n=1, not n=4.
- **Market base rate (from search).** Manual trackers lose 30–40% of applications per published studies; the average active job seeker spends ~15 minutes a day maintaining a spreadsheet. This suggests the class of pain is real at population scale, even if this specific user hasn't hit the acute version yet.
- **Crowded category.** Huntr, Teal, Simplify, ApplyArc, JobCopilot already ship trackers. The user is not opening a new category — they are betting they can build a more personal / better-fit version for themselves, with a differentiated direction (inbox-as-truth + LLM fit-score) that current incumbents do not nail well.

## Status Quo

Current user workflow:

1. LinkedIn / Indeed / company career pages for discovery. Tabs pile up.
2. Gmail for all recruiter correspondence. Threads scatter; "wait, did that company reply?" is a real lived experience.
3. Calendar entries for interviews, often without context linked back to the application.
4. A document or note for role-specific prep / notes.

The failure mode: reconciling across these is itself a part-time job. Applications get lost, follow-ups get forgotten, and there is no coherent picture of "how is my search actually going."

## Target User & Narrowest Wedge

**Target user (v1):** the user themselves. 1 user.
**Target user (v2):** up to 2 close friends willing to tolerate early jank. 3 users total.
**Target user (anywhere beyond v2):** deliberately out of scope for this design.

**Narrowest wedge (v0.1, now superseded):** Job pull from RSS/Greenhouse + CRUD tracker with LLM fit-score.

**Narrowest wedge (v0.2, chosen):** **Target-company polling.** The user curates a list of 20–50 target companies across mixed buckets (big-tech MNC, NYC startup, SG tech, etc.). The app knows how to fetch jobs from each company's ATS (Greenhouse / Lever / YC Work at a Startup / Meta careers JSON / Google careers JSON — tiered by coverage difficulty; see Recommended Approach). Every day (or on-demand), the app polls all target companies, dedupes, LLM-scores fit against the user's profile, and surfaces a filtered view: *new this week, matched to me, grouped by bucket*. Tracker and follow-up drafter remain in v1 as before, but are now downstream of discovery, not co-equal with it.

**Narrowest wedge (v0.3, extended):** **Target-company polling + cold-outreach pipeline.** Same as v0.2, plus a parallel path for companies that don't publicly list the user's target roles. Companies can be tagged `ats: cold-outreach` (or `also_cold_outreach: true` on companies with public listings). For each such company, the user maintains a list of named recipients (people to email). The app drafts a 3–4-sentence cold email per recipient, tracks outreach state through a dedicated state machine, and surfaces a "who haven't I reached out to yet" view. After a positive reply, the outreach path rejoins the standard application state machine, so there is one pipeline of records, not two.

**Narrowest wedge (recommended, not chosen):** Inbox-as-truth extractor — read Gmail, build the tracker from extracted state. The user was pushed on this by the lead advisor and by an independent second opinion, and chose to defer it. This is the single largest risk in the design (see Open Questions).

## Constraints

- **Builder:** solo student at NUS. Limited time budget. Realistic scope = weekends + evenings, not full-time.
- **Users:** 1 (self) in v1; up to 3 in v2. No public distribution.
- **Budget:** LLM API costs are a real concern at student scale. Inference must be bounded (e.g., only re-score on ingest, not on every page load).
- **Infrastructure:** laptop-local is the default. No hosting budget assumed. If it runs locally and serves `http://localhost:8000`, that's a win.
- **OAuth:** Gmail OAuth deferred to v2 (when inbox parsing and auto-send are added). Google's "restricted" scope for sending email is the single biggest deferred risk.
- **Privacy:** v1 reads public job-posting data only. Profile document lives on user's local disk. No email contents touched in v1, so privacy surface is near-zero.

## Premises

**P1. Email is the source of truth for application state, not job posts.** The front of the funnel (the posting) is public; the state of the funnel (screener, onsite, offer, ghost) lives exclusively in email threads. Existing Chrome-extension trackers capture the front and leave the rest to manual entry. Deferred-but-agreed: this is the v2 wedge.

**P2 (revised, per cross-model review).** Auto-ingest matters because manual tracking loses applications silently. The reason is *completeness* — catching the application you'd otherwise forget — not *laziness*. At n=1–3 users, compliance with manual entry may be high enough that auto-ingest is not yet urgent; this is testable.

**P3. Analytics emerges once data is structured.** Once we have `{company, role, status, timestamps, channel, fit-score}`, reply rate / ghost rate / funnel leak are trivial SQL queries. An LLM layer on top ("chat with your search") is thin. No separate analytics product is needed in v1.

**P4. v1 is strictly solo (self). v2 = self + 2 friends with jank tolerance.** No multi-tenant, no public signup, no billing, no polish-for-strangers in v1 or v2.

**P5. OAuth is the hard part, not the LLM.** The LLM parsing is trivial in 2026. Gmail OAuth — especially the write scope needed for auto-send — is Google's most restricted tier and requires app verification. Budget accordingly when v2 is scoped.

**P6. Privacy is architectural, not a feature.** Email contents are sensitive. If emails get processed, the architecture must make the processing location (local vs. cloud LLM) explicit. v1 avoids this entirely by not touching email.

**P7. The product direction is an agent; v1 is honestly a tool.** An agent does work on the user's behalf. A tool stores data. v1 is 95% tool (manual state clicks, clipboard-only drafts, LLM is used for fit-scoring and draft text generation but doesn't act autonomously). v2 moves toward agent (LLM-extracted state transitions from email, approved-send follow-ups). v3+ is genuine agent (discovery automation, autonomous follow-ups with stricter safety gates). Keeping the direction explicit prevents v1 from accidentally ossifying as "yet another tracker."

**P8. Agent safety is first-class.** Follow-up drafting is OK. Follow-up sending without human approval is not. Any agent action that touches the outside world must be gated by explicit user approval in v1 and v2.

**P9. The three capabilities (pull / track / send) do not have equal priority.** They must be sequenced. v1 = pull + track. v2 = email ingest + drafting-with-send. v3 = discovery automation / agent autonomy. Do not try to ship all three at once.

## Approaches Considered

### Approach A — Weekend shipper (minimal)
Static curated list of job sources (a few RSS + Greenhouse ATS for select companies) + simple CRUD tracker + one HTML page. Python FastAPI + SQLite + Jinja. No auth. Human: ~2 weeks; CC: ~6–8 hours. Ships fast, but very commodity; inbox parsing v2 would require partial rewrite. Completeness: 4/10.

### Approach B — Pluggable architecture, inbox-ready (Recommended)
Pluggable `IJobSource` interface. v1 ships exactly two sources: `rss-feed` (generic Greenhouse/Lever/career-page RSS) and `manual-url-paste` (user pastes job URL; LLM extracts `{company, role, jd_text}`). LLM fit-score per job against a user-authored `profile.md`. Tracker with explicit state machine (below) and staleness detection. An `IEventSource` interface is reserved for email events in v2 — designed with the understanding that premature interface design is risky; the interface is intentionally minimal and expected to evolve. Includes "generate follow-up draft" (clipboard-only, no Gmail write scope needed). Python FastAPI + HTMX + SQLite (WAL mode) + Pydantic + Claude API. Human: ~3–4 weeks; CC: ~12–16 hours. Completeness: 8/10.

**Note on the Approach B vs Approach C overlap:** Approach B includes LLM fit-scoring, which is mechanically similar to Approach C's shortlist. The difference is user-facing: B is a browsable tracker where fit-scores help rank a list you explore; C delivers a curated Monday email and asks you not to explore at all. B was chosen because the user said "pull + track" — both activities are first-class, not just the pull.

### Approach C — Weekly brief (lateral)
Reverse the flow. User writes a one-page preferences brief. Every Monday, a cron job generates a personalized shortlist (5–10 jobs) matched to the brief via LLM, with one-line "why this fits you" per job. Tracker is a thin logger around the shortlist. Optionally: ships as email, zero UI. Python cron + Claude API + SQLite + one HTML page (or none). Human: ~2 weeks; CC: ~6–10 hours. Interesting framing (job concierge, not tracker) but a meaningful narrative pivot from what the user described. Completeness: 7/10.

## Recommended Approach

**Approach B — Pluggable, inbox-ready.** Chosen by the user.

Rationale: delivers a real differentiated v1 (LLM fit-score is genuinely non-commodity at 2026 incumbents' quality bar), keeps scope constrained (single-user local app), and does not paint the architecture into a corner for v2. The "zero rewrite" claim for v2 is aspirational — in practice, adding inbox-driven state transitions will require some merging with the manual-click path. "Additive, not rewrite" is the realistic target; the interface design is consciously minimal to reduce the premature-abstraction tax.

Key implementation notes:

- **Job sources are ATS-centric plugins, not feed-centric.** The user maintains a `target_companies.yml` file listing their ~20–50 target companies, each tagged with its ATS type and a config blob (e.g., `{company: "Stripe", ats: "greenhouse", config: {board_slug: "stripe"}, buckets: ["nyc-startup", "big-tech"]}`). On ingest, the app dispatches each company to the correct ATS plugin. v1 ships these plugins in tiers by coverage and effort:

  **Tier 1 (ship in v1, highest ROI):**
  - `greenhouse` — `https://boards-api.greenhouse.io/v1/boards/{board_slug}/jobs`. Covers most YC-backed startups, Stripe, Anthropic, Figma, Notion (alumni), thousands more. Public, free, fast.
  - `lever` — `https://api.lever.co/v0/postings/{company_id}`. Covers another large slice of startups.
  - `waas` — YC Work at a Startup. Public board with job-level pages; Tier 1 if RSS available, else light scraping. Critical for NYC + remote startup coverage.
  - `manual-url-paste` — the catch-all. User pastes any URL, LLM extracts `{company, role, jd_text}`. Always works.

  **Tier 2 (add in v1 if time permits, else early v1.x):**
  - `meta` — `meta.com/careers` exposes a JSON endpoint discoverable via browser DevTools; parse-able with `httpx`.
  - `google` — `careers.google.com` has a public JSON endpoint.
  - `amazon`, `apple`, `microsoft` — each has a public careers JSON endpoint with a bit of scraping.

  **Tier 3 (deferred to v1.x or v2; complex):**
  - `workday` — covers TikTok/ByteDance, many MNCs. JS-rendered; needs Playwright + per-company config. The design doc's earlier deferral of Playwright applies here: a workday plugin is a scope expander, not a v1 necessity.
  - `nodeflair` — Singapore-specific. Scraping required; schema to be worked out.
  - `mycareersfuture` — SG government board. Has a public API but undocumented; research before committing.

- **Target-company discovery (LLM-assisted).** New v0.2 feature: user writes their profile + a short "I want to work at X-type companies in Y locations" brief. An endpoint (`POST /suggest-companies`) sends this to Claude Sonnet 4.6 with a grounded prompt: "Given this profile, suggest N companies the user should add to their target list, grouped by bucket, with one-line reasoning and the likely ATS for each." User reviews suggestions and accepts/rejects into `target_companies.yml`. This turns "I don't know which companies to track" from a blocker into a 2-minute interaction. Limit: suggestions are ungrounded (no web search in v1); user validates against reality.
- **Per-bucket filtering in the UI.** Dashboard lets user filter "new jobs this week" by bucket (big-tech MNC / NYC startup / SG tech / etc.). Makes the mixed-portfolio use case navigable instead of a flat firehose.

- **Cold-outreach pipeline (v0.3).** Any target company can be tagged `ats: cold-outreach` (or `also_cold_outreach: true` when the company also has a public ATS). The app skips ATS polling for cold-outreach-only companies and instead surfaces them in a dedicated UI section: *"Companies to email."* For each cold-outreach target, the user adds one or more `recipients` (name, role, email) — entered manually from LinkedIn or company pages. Per recipient, the user can generate a cold-email draft (see "Cold-email drafter" in the LLM Usage Spec), copy it, and send from their own Gmail. The app logs the outreach as a `cold_outreach_sent` transition on an application row with `source_channel = "cold-outreach"` and `recipient_id` set. Staleness rules apply: a recipient not replied-to within 10 days gets a "nudge" badge; within 14 days auto-moves to `cold_ghost`. If the recipient replies positively, the user manually transitions the application to `cold_reply_yes_intro_call` and the standard application state machine takes over from there — converging the two paths into one record set.
- **Profile as a markdown file.** Power-user escape hatch: data is the interface. User edits `~/.config/job-tracker/profile.md` directly; the app reads it on every ingest. No settings UI in v1.
- **Fit-score is cached.** Claude Haiku 4.5 called once per new job at ingest. Result cached in SQLite. Re-scoring is triggered by either a profile-file mtime change (auto-detected on ingest) or an explicit `POST /rerank` endpoint (callable via a "re-rank all" button in the UI).
- **State machine is explicit in v1.** User clicks through states manually. See Data Model section below for the full transition graph. v2 adds an optional LLM-inferred transition path alongside the manual path; the manual path remains the source of truth and override.
- **Follow-up draft generator is in v1** (and is an explicit scope decision: draft-only, no Gmail OAuth, no send). Inputs: tracker state, last contact date, stage, company context, optionally the pasted JD. Output: draft text copied to clipboard via `navigator.clipboard.writeText()` in the browser (localhost origins support this without extra permission). Degrades to a "copy this" textarea if clipboard API is blocked.
- **Single-user in v1.** No login, no auth. App binds to `127.0.0.1:8000` only. Profile at `~/.config/job-tracker/profile.md`.

### Data Model

SQLite schema (v1):

```sql
-- target_companies: user's curated list; populated from target_companies.yml
CREATE TABLE target_companies (
    id              TEXT PRIMARY KEY,           -- slug; stable
    display_name    TEXT NOT NULL,              -- 'Stripe', 'TikTok', etc.
    ats_type        TEXT NOT NULL,              -- 'greenhouse' | 'lever' | 'meta' | 'google' | 'workday' | 'waas' | 'nodeflair' | 'manual'
    ats_config      TEXT NOT NULL,              -- JSON blob, varies by ats_type
    buckets         TEXT NOT NULL,              -- JSON array: ['big-tech', 'nyc-startup', 'sg-tech', ...]
    added_at        TEXT NOT NULL,
    last_polled_at  TEXT,
    last_poll_error TEXT                        -- nullable; human-readable failure reason from last fetch
);

-- jobs: one row per unique job posting ingested from any source
CREATE TABLE jobs (
    id              TEXT PRIMARY KEY,           -- sha1(canonical_url or company+role+source)
    source          TEXT NOT NULL,              -- ats_type, or 'manual-url-paste'
    source_url      TEXT NOT NULL,
    canonical_url   TEXT,                       -- after URL-normalization
    company_id      TEXT REFERENCES target_companies(id),  -- nullable for manual-paste from untracked companies
    company         TEXT NOT NULL,              -- denormalized for non-tracked companies
    role            TEXT NOT NULL,
    jd_text         TEXT,                       -- full JD markdown, extracted by LLM if needed
    location        TEXT,                       -- free-form string; queried via LIKE
    locations_norm  TEXT,                       -- JSON array of normalized tags: ['nyc', 'sg', 'remote-us', 'remote-apac']
    posted_at       TEXT,                       -- ISO8601, may be null
    ingested_at     TEXT NOT NULL,
    fit_score       INTEGER,                    -- 0–100, NULL until scored
    fit_reason      TEXT,                       -- 1-line LLM rationale
    profile_mtime   TEXT                        -- profile mtime when scored; re-score if stale
);

-- applications: one row per application the user has made
CREATE TABLE applications (
    id              TEXT PRIMARY KEY,           -- uuid
    job_id          TEXT REFERENCES jobs(id),   -- nullable: apps from before v1 may have no job row
    company         TEXT NOT NULL,              -- denormalized for pre-v1 applications
    role            TEXT NOT NULL,
    applied_at      TEXT NOT NULL,
    current_state   TEXT NOT NULL,              -- see state machine below
    last_contact_at TEXT,                       -- updated on state transition
    source_channel  TEXT,                       -- 'referral' | 'direct' | 'recruiter' | 'job-board'
    notes_md        TEXT                        -- free-form user notes
);

-- transitions: append-only log of state changes (auditability + future LLM training)
CREATE TABLE transitions (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    application_id  TEXT NOT NULL REFERENCES applications(id),
    from_state      TEXT,                       -- null for initial 'applied' or 'cold_outreach_sent'
    to_state        TEXT NOT NULL,
    occurred_at     TEXT NOT NULL,
    evidence        TEXT,                       -- free-form: email subject line, user note, etc.
    inferred_by     TEXT                        -- 'user' in v1; 'llm' in v2
);

-- recipients (v0.3): named contacts for cold outreach. One company can have multiple recipients.
CREATE TABLE recipients (
    id                  TEXT PRIMARY KEY,        -- uuid
    company_id          TEXT NOT NULL REFERENCES target_companies(id),
    name                TEXT NOT NULL,
    role                TEXT,
    email               TEXT,                    -- nullable: user may have only a LinkedIn URL initially
    linkedin_url        TEXT,
    notes               TEXT,                    -- why this person, how found
    added_at            TEXT NOT NULL,
    last_contacted_at   TEXT,
    last_reply_at       TEXT,
    status              TEXT NOT NULL DEFAULT 'not_contacted'  -- 'not_contacted' | 'contacted' | 'replied_yes' | 'replied_no' | 'ghost'
);

-- applications table gets a nullable recipient_id for cold-outreach rows
-- (add to existing table definition above, or via migration):
--   recipient_id TEXT REFERENCES recipients(id)  -- nullable; set when source_channel = 'cold-outreach'
```

Note: `applications.source_channel` values in v0.3 include `'referral'`, `'direct'`, `'recruiter'`, `'job-board'`, and `'cold-outreach'`. The `cold-outreach` variant requires a non-null `recipient_id`.

**Dedup strategy.** Jobs are identified by a hash of `canonical_url` (URL-normalized: strip tracking params, lowercase host, remove trailing slash). If `canonical_url` is not available (some RSS feeds), fall back to `sha1(company + '|' + role + '|' + source)`. On ingest, check `jobs.id` before inserting. Two feeds surfacing the same posting results in one row, with the earlier `ingested_at` retained.

### State Machine

States (v1 has all of these; `ghost` is time-derived, not user-set):

```
NEW_APPLICATION
  applied
  ├─ screener_scheduled
  │   └─ screener_done
  │       ├─ technical_scheduled
  │       │   └─ technical_done
  │       │       ├─ onsite_scheduled
  │       │       │   └─ onsite_done
  │       │       │       ├─ offer
  │       │       │       │   ├─ offer_accepted
  │       │       │       │   ├─ offer_declined
  │       │       │       │   └─ offer_negotiating
  │       │       │       └─ rejected_post_onsite
  │       │       └─ rejected_post_technical
  │       └─ rejected_post_screener
  ├─ rejected_no_response_short   (auto: applied + 14d, no reply)
  ├─ withdrawn_by_candidate        (user-set at any stage)
  └─ ghost                         (auto: last_contact_at + 14d with no user action, excluding rejected/withdrawn/accepted)
```

Not every transition is linear — a company may skip `screener` and go straight to `technical`. The state machine is permissive (any state can transition to `withdrawn_by_candidate`, `rejected_*`, or manually to any other valid next-state). v2 may add `phone_screen_scheduled` and multi-round onsite states if needed.

**Cold-outreach subgraph (v0.3)** — alternate entry point; converges back into the main machine after a positive reply:

```
COLD_OUTREACH_ENTRY
  cold_outreach_sent
    ├─ cold_reply_yes_intro_call       → (then applied, screener_scheduled, etc. as usual)
    ├─ cold_reply_referred             → new cold_outreach_sent on referred contact (fresh application row)
    ├─ cold_reply_no_opening           (terminal; user can re-send after 90d via a "re-engage" button)
    └─ cold_ghost                      (auto: sent + 14d, no reply)
```

The `cold_reply_yes_intro_call` state acts as the convergence point. From that state onward, the application uses the standard state machine transitions (onsite_scheduled, offer, etc.). This means one application record can carry the full history from first cold email through offer — no separate "outreach" and "application" silos.

### Plugin Interface Sketch

```python
# v1 interface, intentionally minimal
class IJobSource(Protocol):
    name: str  # 'rss-feed' | 'manual-url-paste' | ...
    async def fetch(self, config: SourceConfig) -> list[RawJob]:
        """Return a list of RawJob dicts. May be empty. Raises on config error; not on network fail."""

class SourceConfig(TypedDict, total=False):
    # rss-feed config: `urls` is a list of RSS endpoints to poll
    urls: list[str]
    # manual-url-paste config: `url` is the single URL to scrape on demand
    url: str
    # reserved for future sources: max results per fetch
    limit: int

class RawJob(TypedDict):
    source_url: str
    company: str
    role: str
    jd_text: str | None           # may be empty; LLM fills in later
    location: str | None
    posted_at: str | None         # ISO8601 or None

# v2 sketch (not implemented in v1; reserved to force design-for-future)
class IEventSource(Protocol):
    name: str  # 'gmail-imap' | 'gmail-api' | ...
    async def fetch(self, since: str) -> list[RawEvent]:
        """Return events (e.g., inbound emails) since the given ISO8601 timestamp."""

class RawEvent(TypedDict):
    source: str
    occurred_at: str
    subject: str
    body: str
    from_email: str
    # v2 is where state-inference-from-event lives; interface expected to evolve
```

### profile.md Schema (example)

The file is freeform markdown but the LLM is prompted to parse specific sections. Users can deviate; the prompt instructs the LLM to use its best judgment on non-standard formats.

```markdown
# Profile

## Roles I want
- Data engineer / ML engineer / platform / backend
- Python or Go preferred; open to TypeScript

## Company stage
- Series A–C, product-led, 50–1000 headcount
- Open to later-stage if the team is strong

## Location
- Singapore, remote-APAC, hybrid-SG
- No US relocation in 2026

## Avoid
- Defense, crypto speculation, ad-tech
- Companies with <18 months runway based on last public round

## Compensation
- SGD 90k+ base
- Open on equity

## What I care about
- Technical depth of the team
- Code review culture
- Clear product thesis
```

### Staleness Rules

- **Stale warning (dashboard badge only):** `last_contact_at + 10 days` passed with no user action on an application in an active state (not `rejected_*`, `withdrawn_*`, `offer_*`, `ghost`).
- **Auto-ghost:** `last_contact_at + 14 days` passed and no user transition → automatic transition to `ghost`. User can manually override back to prior state.
- **No email notifications in v1.** Staleness is surfaced only in the local app UI (a badge on the dashboard and a sorted "stale" filter). v2 may add a weekly digest.
- **Trigger:** computed on every dashboard page load (cheap SQL). No background cron in v1.

### LLM Usage Spec

**Fit-score prompt (Haiku 4.5):**

- **Input:** `profile.md` contents + job `{company, role, jd_text}`.
- **Output:** strict JSON `{score: int 0–100, reason: str max 180 chars}`.
- **Temperature:** 0.2 for consistency.
- **Rubric (in the prompt):** 100 = direct match on role + tech + location + company-fit (stage/values extracted from profile.md "Company stage" section); 70 = 3 of 4 match; 40 = partial match with clear mismatches; 0 = explicit avoid-list hit. The LLM infers stage/values from the job's company name + jd_text; no dedicated `stage` column in `jobs` (kept denormalized-in-prompt to keep v1 simple).
- **Fallback if accuracy bar (Success Criterion #2) not met:** iterate in this order: (1) enrich profile.md with examples of past jobs you liked/didn't like, (2) switch scoring model to Sonnet 4.6 for difficult ranks, (3) add a per-job "why was this match wrong" feedback loop that appends to the profile over time.

**Follow-up draft prompt (Sonnet 4.6):**

- **Input:** application row + last few transitions + pasted JD + optional user tone hint ("warm" / "neutral" / "direct").
- **Output:** plain text draft, 80–150 words, no subject line (user adds).
- **Temperature:** 0.5.

**Cold-email draft prompt (Sonnet 4.6, v0.3):**

- **Input:** user profile + company name + recipient `{name, role, notes}` + optional "why this company" note (short, in user's own words) + target ask ("intro call" / "referral" / "exploring roles").
- **Output:** plain text draft, **60–100 words** (deliberately tighter than follow-ups; cold emails that exceed ~100 words are read as noise). No subject line. Structure: (1) one-sentence specific hook about the company or recipient's work, (2) one sentence about you, (3) one-sentence ask, (4) one-line out.
- **Temperature:** 0.6 (slightly higher than follow-ups — more variation across recipients prevents form-letter feel).
- **Guardrails in the prompt:** no "I hope this email finds you well," no "I came across your profile and was impressed," no generic praise. If the "why this company" note is empty, the LLM asks the user to add one (returned as `{error: "needs_why_note"}`) rather than generating a generic pitch. This forces the user to have an actual reason, which is the whole point of cold outreach working.

### Operational Concerns

- **SQLite in WAL mode.** `PRAGMA journal_mode=WAL;` at startup. Removes read-write locking surprises if a background poller is added later (not in v1 — ingestion is CLI-triggered via `python -m job_tracker ingest`).
- **Missing or malformed `profile.md`.** On startup, if the file does not exist, create it with a templated stub (the example in the profile.md Schema section above). If the file parses as empty, fit-scoring is skipped and every job gets `fit_score = NULL` with a UI banner reading "write your profile to enable fit-scoring — edit ~/.config/job-tracker/profile.md". No crash, no partial state.
- **Schema migrations.** v1 does not include a migration framework (Alembic etc.) to stay lean. v2 will either add Alembic or ship a one-off `migrate()` function that branches on a `PRAGMA user_version`. Expected v2 schema changes: add `events` table for email-derived events, add `inferred_by='llm'` rows in `transitions`. The `current schema_version` is recorded in SQLite via `PRAGMA user_version = 1` at v1 init.
- **Canonical URL normalization.** Use `w3lib.url.canonicalize_url` (standard Python library used by Scrapy). Additionally strip tracking params: `utm_*`, `gh_src`, `ref`, `refid`, `source`, `mc_eid`, `fbclid`, `gclid`. Lowercase host. Remove trailing slash. Sort remaining query params alphabetically.
- **Clipboard API on Safari.** `navigator.clipboard.writeText()` requires a secure context. `http://127.0.0.1` qualifies on Chrome/Edge/Firefox but Safari treats it as non-secure and may reject the call. Detection: try the clipboard API; on failure, fall back to a visible `<textarea>` with an auto-select + copy-instruction. Behavior degrades, does not break.
- **Claude API failures.** Retry with exponential backoff (3 attempts). On final failure, persist the job row with `fit_score = NULL` and surface a "unscored — retry" button in the UI. Never block ingestion on scoring.
- **RSS fetch failures.** Log and skip. Do not fail the entire ingest. Next ingest retries.
- **JS-rendered career pages** (Workday, Greenhouse embeds, LinkedIn job pages). `httpx` fetches an empty shell. v1 mitigations: (a) for Workday/iCIMS/Greenhouse-embed, known URL patterns can be rewritten to their API/JSON endpoints; (b) where no workaround exists, fall back to "LLM extracts from the URL and title alone" with a warning badge "JD text unavailable." A headless browser (Playwright) is explicitly deferred; it adds a ~300MB dependency and is not justified for v1.
- **Cost model (refined).** At 100 new jobs/week: Haiku fit-score ~$0.30/month. Follow-up drafts on Sonnet at ~5 drafts/week = ~$0.20/month. Total: well under $1/month. URL-paste JD extraction (Haiku): ~$0.10/month. These are back-of-envelope; empirically verify after v1's first week.

5. **Storage location of profile.md.** In-repo, in `~/`, or in a config dir. Matters for v2 multi-user. Recommendation: v1 reads from `~/.config/job-tracker/profile.md` (standard xdg-style). Multi-user in v2 namespaces per user.

## Success Criteria

**v1 is successful if:**

1. The user dogfoods it on their own active job search for at least 2 weeks without reverting to LinkedIn-as-primary. "Did I actually build something I want to use."
2. **Discovery metric (v0.2):** the tool surfaces at least 5 jobs per week that the user had *not* seen on their own via LinkedIn / company sites. This validates that company-centric polling is actually catching things the user would have missed. Tracked via a `saw_before_tool: bool` tag the user sets when reviewing the weekly digest.
2b. **Cold-outreach metric (v0.3):** at least 5 cold emails sent through the drafter in the first 2 weeks of dogfooding, with at least one generating a positive reply (intro call booked OR referral received). If zero replies after 10 sent emails, the drafter's prompt or the recipient-selection strategy needs revision.
3. The LLM fit-score is accurate enough to be actionable. Operational definition: after 4 distinct ingest sessions (each yielding ≥5 new jobs), the user records agreement with the ranked top-3 via a simple `POST /apps/{id}/rating` endpoint (body: `{rating: "agree"|"disagree", notes: str}`) that appends to a `ratings` table. If ≥75% of the top-3 ratings across the first 20 rated jobs are `agree`, the bar is met. If not, iterate per the fallback ladder in the LLM Usage Spec.
4. At least one application is caught / recovered via the staleness-alert feature that the user would have otherwise missed. This validates the "data integrity" half of the premise.
5. The architecture admits an email-ingest plugin in v2 as a primarily-additive change. Operational definition: v2 ships with ≤30% of its diff being edits to existing v1 files, and the existing manual-transition code path remains intact as the fallback / user-override. "Zero rewrite" is aspirational; "additive, not rewrite" is the real target.

**v1 is NOT about:** scaling, supporting friends, polish, public users, revenue, or winning a competitive bake-off against Huntr/Teal.

## Distribution Plan

- **v1:** local-only. Runs on user's laptop via `uvicorn main:app`. Binds `127.0.0.1:8000`. Zero distribution. No install for anyone but the author.
- **v2 (self + 2 friends)** — detailed elsewhere, summarized: each friend gets their own deployed instance (not shared). Single-user-per-deploy is the simplest multi-user story: no auth code in the app, hosting boundary = user boundary. Each friend runs their own Fly.io or Render instance (free tier), with a shared deployment recipe you maintain. Per-user Gmail OAuth (v2) uses Google's "testing" mode with each friend added as a test user (max 100, far from binding). Privacy story: data never crosses friend-boundaries because instances are isolated. This is explicitly janky; picking it over "real multi-tenant" is the whole point of "friends who tolerate jank."
- **v3+ (anything broader):** out of scope for this design. Would require: Google OAuth app verification, privacy policy, real auth, shared multi-tenant data isolation, hosting budget. Do not build any of this speculatively in v1 or v2.
- **No CI/CD pipeline in v1.** Manual deploy if any. A v2 multi-instance deployment would warrant a single GitHub Actions workflow that each friend can trigger against their own Fly.io / Render app.

## Dependencies

- **Runtime:** Python 3.11+, FastAPI, uvicorn, SQLite (stdlib), Pydantic v2, HTMX (via CDN), Jinja2 (for the few Jinja templates).
- **LLM:** Anthropic Python SDK. Haiku 4.5 for fit-scoring (cheap, fast; ~$0.30/month at 100 jobs/week). Sonnet 4.6 for follow-up drafts (higher quality for outbound text; ~$0.20/month at 5 drafts/week).
- **Ingestion:** `feedparser` for RSS, `httpx` for URL fetch.
- **Explicitly NOT in v1:** Playwright / headless browser (deferred; see Operational Concerns), any database server (no Postgres, no MySQL), any queue (no Redis, no Celery), any auth library (no users in v1).
- **SQLite PRAGMAs set at startup:** `journal_mode=WAL`, `synchronous=NORMAL`, `foreign_keys=ON`.
- No external services required in v1 beyond Anthropic API.
